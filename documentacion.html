Panel del Cajero — Banca Digital
Arquitectura Modular ES6 · POO · Event Loop

📁 13 módulos JavaScript 🎨 1 hoja de estilos CSS 📄 1 archivo HTML (SPA) 🔗 Sin dependencias externas

1. Resumen del Proyecto

Mi Plata es una aplicación web de banca digital que simula un panel de cajero. Permite al usuario consultar saldos, realizar consignaciones, retiros, transferencias, pagar servicios, hacer recargas, comprar a cuotas con tarjeta de crédito y gestionar su perfil.

El proyecto nació como un único archivo HTML de más de 2.000 líneas. Fue completamente refactorizado a una arquitectura modular ES6, separando responsabilidades en 13 módulos independientes sin perder ninguna funcionalidad visible.

Objetivo académico: demostrar el uso de Programación Orientada a Objetos (POO), módulos ES6, el modelo del Event Loop de JavaScript y buenas prácticas de organización de código frontend sin frameworks ni bibliotecas externas.

2. Arquitectura General

2.1 Single Page Application (SPA)

La aplicación utiliza un único index.html como contenedor. Todas las "páginas" (Dashboard, Perfil, Movimientos, Extractos) son <div> dentro del mismo documento, que se muestran u ocultan dinámicamente mediante JavaScript. No hay recarga de página al navegar.

¿Por qué una SPA? Porque el estado (usuario logueado, saldo, historial) vive en memoria JavaScript. Con múltiples archivos HTML habría que pasar ese estado por localStorage entre páginas, lo cual es más complejo y propenso a errores.

2.2 Módulos ES6

Cada archivo .js usa la sintaxis import / export nativa del navegador. Esto obliga a servir los archivos desde un servidor HTTP (no funciona con file://) pero permite al navegador resolver dependencias automáticamente, hacer lazy loading y garantizar que cada módulo se ejecuta solo una vez.

2.3 Grafo de Dependencias (sin ciclos)

Un principio fundamental de la arquitectura es que el grafo de importaciones es un DAG (Directed Acyclic Graph), es decir, no hay dependencias circulares. Un módulo A que importa B no puede hacer que B importe A.

app.js importa a todos los demás módulos

models.js ← (no importa nada del proyecto)
state.js ← models.js
utils.js ← (no importa nada del proyecto)
toast.js ← utils.js
modal.js ← utils.js
router.js ← utils.js, state.js
render.js ← utils.js, state.js, modal.js, toast.js
auth.js ← state.js, router.js, render.js, toast.js
documentos.js ← utils.js, state.js, modal.js, render.js
perfil.js ← utils.js, state.js, modal.js, toast.js
transacciones.js ← utils.js, state.js, models.js, modal.js, toast.js, render.js, documentos.js
servicios.js ← utils.js, state.js, models.js, modal.js, toast.js, render.js, documentos.js
Caso especial: render.js necesita abrir recibos (definidos en documentos.js), pero no puede importarlo porque documentos.js importa a render.js — eso crearía un ciclo. Se resolvió con el patrón inyección de callback: render.js expone setMovClickHandler(fn) y app.js inyecta la función abrirRecibo desde afuera.

3. Módulos — Detalle Técnico

models.js Clases POO

Define toda la jerarquía de objetos del dominio bancario. Es el módulo más "puro": no importa nada del proyecto y no tiene efectos secundarios.

Jerarquía de herencia

Cuenta (clase base abstracta)
  ├── CuentaAhorros
  └── CuentaCorriente
TarjetaCredito extends Cuenta
Cliente

Clase Movimiento

Representa cada transacción. Se crea con new Movimiento(tipo, monto, descripcion, meta). El constructor asigna automáticamente un id único y la fecha actual.

Clase Cuenta (abstracta)

Define la interfaz común: numero, titular, saldo (campo privado #saldo con getter/setter que lanza error si el valor es negativo), el array movimientos[] y los métodos registrarMovimiento(m) y consignar(monto, desc).

¿Por qué #saldo privado? Para garantizar que nadie asigne un saldo negativo por accidente. El setter valida antes de asignar. Es encapsulamiento real en JavaScript moderno (ES2022).

Clase CuentaAhorros

Hereda de Cuenta. Sobreescribe retirar(monto): aplica un interés del 1.5% sobre el monto retirado (interés de captación simulado). Lanza error si el monto supera el saldo.

Clase CuentaCorriente

Hereda de Cuenta. Calcula un limiteRetiro del 120% del saldo (sobregiro del 20%). Permite retirar hasta ese límite, marcando el movimiento como sobregiro: true cuando aplica.

Clase TarjetaCredito

Hereda de Cuenta pero no usa el campo saldo. En su lugar tiene cupo (límite total) y deuda (monto usado). El getter cupoDisponible = cupo − deuda.

  • comprar(monto, cuotas, desc): valida cupo, incrementa deuda, calcula cuota mensual con fórmula de anualidad.
  • static tasa(cuotas): 0% para ≤2 cuotas, 1.9% para 3–6, 2.3% para ≥7.
  • static cuotaMensual(capital, cuotas): aplica la fórmula financiera C·i / (1−(1+i)^−n).

Clase Cliente

Almacena los datos personales: identificacion, nombre, celular, correo, direccion, foto, usuario, password. Tiene el método agregarCuenta(cuenta) que mantiene el array cuentas[].

state.js Estado Compartido

Crea las instancias demo del modelo y exporta el objeto mutable ui que todos los módulos comparten por referencia.

¿Por qué un objeto ui compartido?

En lugar de pasar parámetros entre funciones o usar variables globales en window, todos los módulos importan el mismo objeto ui. Como JavaScript pasa objetos por referencia, cuando un módulo muta ui.cuentaActiva, todos los demás ven el cambio inmediatamente sin necesidad de un sistema de eventos. 

export const ui = {
  cuentaActiva:   ahorros,   // cuenta seleccionada actualmente
  saldoVisible:   true,      // toggle del ojo en el dashboard
  extractoFiltro: 'todos',   // filtro activo en la página de extractos
  intentosLogin:  0,         // contador de intentos fallidos
};

El array clientes = [cliente, otroCliente] permite hacer búsquedas por cédula en consignaciones y transferencias a terceros.

utils.js Utilidades y Scheduler

Helpers del DOM

export const $ = (sel) => document.querySelector(sel);
export const $$ = (sel) => [...document.querySelectorAll(sel)];

Atajos para no escribir document.querySelector en cada módulo. El $$ convierte el NodeList en un array real para poder usar .forEach, .map, etc.

Función fmt(n)

Formatea un número como moneda colombiana: fmt(2450000) → $2.450.000. Usa la API nativa Intl.NumberFormat.

El Scheduler — Event Loop por prioridades

JavaScript es single-threaded: ejecuta una tarea a la vez. Pero tiene distintas "colas" de ejecución con diferentes prioridades. El Scheduler expone cada cola como un método simple:

Método Mecanismo interno Prioridad Cuándo usarlo
Scheduler.micro(fn) Promise.resolve().then(fn) 1 (más alta) Actualizar estado lógico antes del siguiente render
Scheduler.frame(fn) requestAnimationFrame(fn) 2 Toda mutación del DOM (sincronizada con el repintado)
Scheduler.defer(fn, ms) setTimeout(fn, ms) 3 Acciones diferidas (ej: abrir recibo 400ms después)
Scheduler.idle(fn) requestIdleCallback(fn) 4 (más baja) Tareas de baja prioridad cuando el navegador está libre
¿Por qué usar requestAnimationFrame para el DOM? El navegador repinta la pantalla a 60 FPS (cada ~16ms). Si mutamos el DOM fuera de ese ciclo, forzamos repintados innecesarios. Con rAF, la mutación queda lista justo antes del próximo frame, eliminando el parpadeo y reduciendo el trabajo del navegador.

El Event Loop de JavaScript (concepto)

El orden de ejecución cuando el usuario hace clic es:

  1. Call stack: la función del event listener corre hasta el final.
  2. Microtask queue: se vacía completamente (Promise.then, queueMicrotask).
  3. Render step: el navegador recalcula estilos, layout y ejecuta callbacks de rAF.
  4. Macrotask queue: se procesa una tarea (setTimeout, eventos de I/O).
  5. El ciclo vuelve al paso 2.

router.js Navegación SPA

Implementa un enrutador del lado del cliente sin biblioteca externa. Cuando el usuario hace clic en un ítem del sidebar, no se carga una nueva página; el router oculta la vista actual y muestra la nueva.

Mapa de vistas

const VISTAS = {
  inicio:      { el: '#viewDashboard',   titulo: 'Inicio'      },
  perfil:      { el: '#viewPerfil',      titulo: 'Mi Perfil'   },
  movimientos: { el: '#viewMovimientos', titulo: 'Movimientos' },
  extractos:   { el: '#viewExtractos',   titulo: 'Extractos'   },
};

Función navegarA(id)

Usa el Scheduler para separar la lógica en dos fases:

  1. Microtask: oculta todas las vistas (DOM write pequeño, sin layout).
  2. Frame (rAF): muestra la vista destino, actualiza el título, marca el botón activo en el sidebar y ejecuta el callback onEnter de esa vista.

Función registrarOnEnter(vistaId, fn)

Registra una función que se ejecuta cada vez que se navega a una vista. Esto permite a cada módulo inicializarse "bajo demanda" en lugar de hacerlo todo al cargar. Por ejemplo, la página de Movimientos solo construye su tabla cuando el usuario navega a ella.

render.js Render del Dashboard

Contiene las funciones que actualizan el DOM del dashboard principal. Todas usan Scheduler.frame para sincronizarse con el ciclo de repintado.

FunciónQué hace
inicializarPanel()Muestra el #app y oculta el login tras autenticarse.
renderTabs()Genera los botones de cuentas (Ahorros, Corriente, Tarjeta) con el saldo de cada una.
renderSaldo()Actualiza la tarjeta principal con el saldo de la cuenta activa. Respeta el toggle de visibilidad.
renderMovimientos()Lista los últimos movimientos en el panel derecho del dashboard.
renderStats()Calcula y muestra los totales del mes (ingresos, gastos, cantidad de movimientos).
setMovClickHandler(fn)Inyecta el callback para abrir el recibo al hacer clic en un movimiento.
Patrón de inyección de callback: render.js necesita llamar a abrirRecibo() (de documentos.js), pero no puede importar ese módulo porque crearía una dependencia circular. En cambio, app.js llama a setMovClickHandler(abrirRecibo) al inicio, inyectando la función desde afuera. render.js la guarda en una variable interna y la usa cuando necesita.

auth.js Autenticación

Maneja el inicio y cierre de sesión. La autenticación es local (sin servidor), comparando usuario y contraseña con los datos del objeto cliente en state.js.

Flujo de login

  1. El usuario envía el formulario #formLogin.
  2. Se compara con cliente.usuario y cliente.password.
  3. Si falla, incrementa ui.intentosLogin y muestra el contador de intentos. Al tercer fallo, bloquea el formulario por 10 segundos.
  4. Si acierta, llama a inicializarPanel() (muestra la app) y navegarA('inicio').

Función cerrarSesion()

Oculta el #app, muestra el login, limpia los campos y resetea el contador de intentos.

modal.js toast.js Componentes UI

modal.js

Controla el único <div id="modal"> del HTML. La función abrirModal({ titulo, sub, body, footer }) recibe objetos de configuración y renderiza el contenido dinámicamente en el modal existente. cerrarModal() lo oculta. El módulo también cierra el modal al hacer clic en el backdrop o al presionar Escape.

¿Por qué un solo modal en el HTML? Crear y destruir elementos DOM es costoso. Con un solo modal que se reutiliza, el navegador no tiene que recalcular el layout repetidamente.

toast.js

Muestra notificaciones flotantes en la esquina inferior derecha. toast(titulo, mensaje, tipo) crea un elemento, lo inserta en el DOM, lo anima con CSS y lo elimina automáticamente después de 3.5 segundos usando Scheduler.defer. Tipos: 'success', 'error', 'warn'.

transacciones.js Operaciones Bancarias

Define las cuatro operaciones principales que abren un modal flotante.

accionConsultar()

Muestra un modal con el saldo de la cuenta activa y los últimos 3 movimientos. Detecta el tipo de cuenta para mostrar información relevante (interés en ahorros, sobregiro en corriente, cupo en tarjeta).

accionConsignar()

Ofrece dos modos mediante un toggle:

  • A mi cuenta: llama a cuentaActiva.consignar(monto), que incrementa el saldo y crea el movimiento.
  • A otra persona: busca al destinatario en el array clientes por cédula (búsqueda en tiempo real), incrementa el saldo de su cuenta y registra el movimiento allí.

accionRetirar()

Para cuentas normales llama a cuentaActiva.retirar(monto), que aplica la lógica de intereses o sobregiro según el tipo. Para Tarjeta de Crédito muestra el modal de Avance de Cajero: incrementa la deuda de la tarjeta con interés del 2.3% mensual.

accionTransferir()

Para cuentas normales: toggle entre "Mis cuentas" (dropdown) y "Por cédula" (lookup). Descuenta de la cuenta origen e incrementa la destino. Para Tarjeta de Crédito muestra el modal de Avance: incrementa la deuda y deposita en la cuenta destino.

servicios.js Servicios y Tarjeta

accionServicios()

Grilla de 6 empresas (EPM, Claro, Tigo, Acueducto, Gas, EPS). El usuario selecciona empresa, ingresa referencia y monto. Descuenta del saldo y crea movimiento de tipo 'servicio'.

accionRecargas()

Grilla de 4 operadores + selector de número celular + botones de monto predefinido ($5.000 a $50.000). Crea movimiento de tipo 'recarga'.

accionComprarTC()

Cambia automáticamente a la tarjeta de crédito si no está activa. Muestra monto, selector de cuotas (1, 2, 3, 6, 12, 24) y un resumen dinámico que calcula en tiempo real la cuota mensual y el total a pagar usando la fórmula de anualidad de TarjetaCredito.cuotaMensual().

documentos.js Páginas Completas y Recibos

Maneja las vistas de página completa (no modales) y la generación de recibos.

initMovimientosPage()

Construye la tabla completa de movimientos de todas las cuentas. Tiene tabs por cuenta y filtro por tipo. Se ejecuta vía el callback onEnter del router cuando el usuario navega a "Movimientos".

initExtractosPage()

Similar a movimientos pero con filtro por tipo de transacción y un resumen estadístico del período.

abrirRecibo(movimiento)

Recibe un objeto Movimiento y genera dinámicamente un modal con formato de recibo bancario: número de operación, fecha, tipo, monto, descripción y metadatos específicos del tipo (cuotas para compras, referencia para servicios, etc.).

Flags de listeners: Las páginas de movimientos y extractos usan variables booleanas (_movListenersAdded, _extListenersAdded) para asegurarse de que los event listeners se agregan solo una vez, aunque el usuario navegue a esas páginas múltiples veces durante la sesión.

perfil.js Página de Perfil

Renderiza la página completa de perfil con los datos del cliente. Permite cambiar la foto de perfil (sube archivo y usa FileReader para mostrarla como Data URL) y cambiar la contraseña con validación inline (sin abrir modal).

El cambio de contraseña valida que la clave actual sea correcta, que la nueva tenga mínimo 4 caracteres y que la confirmación coincida. Actualiza directamente cliente.password en el estado.

app.js Punto de Entrada

Es el único módulo cargado directamente desde el HTML (<script type="module" src="js/app.js">). Su función es conectar todos los demás módulos, no implementar lógica de negocio.

Responsabilidades en orden:

  1. Inicializar infraestructura: initModal(), initLogin().
  2. Inyectar dependencias: setMovClickHandler(abrirRecibo) resuelve la dependencia circular.
  3. Registrar callbacks de navegación: qué hacer al entrar a cada página.
  4. Registrar event listeners globales: barra superior (logout, toggle saldo, menú móvil).
  5. Delegación de clics: un solo listener en document que despacha acciones por data-action.
  6. Dropdowns de la topbar: notificaciones y atajos.
Patrón de delegación de eventos: en lugar de agregar un listener a cada botón del sidebar, se pone uno solo en document. Cuando el usuario hace clic, el evento "burbujea" hasta document, donde se lee el atributo data-action del elemento más cercano y se despacha la acción correspondiente. Esto es más eficiente en memoria y funciona aunque los botones se recreen dinámicamente.

4. Flujos Principales

4.1 Flujo de Login

Usuario envía formulario
  → auth.js valida credenciales contra state.js
  → Si correcto:
      render.js.inicializarPanel()   // muestra #app, oculta #login
      router.navegarA('inicio')
        → Scheduler.micro: oculta vistas
        → Scheduler.frame: muestra #viewDashboard
        → onEnter('inicio'): renderTabs, renderSaldo, renderMovimientos, renderStats

4.2 Flujo de Transacción (ej: Consignar)

Click en "Consignar" en el sidebar
  → app.js captura el clic (delegación)
  → Llama transacciones.accionConsignar()
  → modal.js abre el modal con el formulario
  → Usuario completa y confirma
  → models.CuentaAhorros.consignar(monto) actualiza el saldo
  → cerrarModal()
  → render.renderSaldo(), renderMovimientos(), renderStats()  // refresca UI
  → toast('Consignación exitosa', ...)
  → Scheduler.defer(() => documentos.abrirRecibo(m), 400)   // recibo tras 400ms

4.3 Flujo de Navegación

Click en "Movimientos" en el sidebar
  → data-action="movimientos" detectado en app.js
  → router.navegarA('movimientos')
  → Scheduler.micro: oculta todas las .view
  → Scheduler.frame: muestra #viewMovimientos, actualiza sidebar y título
  → onEnter callback → documentos.initMovimientosPage()
  → Construye la tabla de movimientos en el DOM

5. Decisiones de Diseño

Decisión Alternativa Por qué se eligió esta
Un solo index.html (SPA) Un HTML por página (MPA) El estado en memoria no requiere localStorage; navegación instantánea.
ES6 import/export nativo Bundler (Webpack, Vite) Sin herramientas de build; el navegador resuelve las dependencias directamente.
Un objeto ui mutable compartido Redux / Zustand / EventEmitter Suficiente para la escala del proyecto; sin overhead de una librería de estado.
requestAnimationFrame para DOM Mutaciones síncronas directas Evita layout thrashing y sincroniza con el ciclo de 60 FPS del navegador.
Inyección de callback (setMovClickHandler) Importación directa (circular) Rompe el ciclo de dependencias sin introducir un EventBus externo.
Panel promo con CSS slideshow Librería de carrusel (Swiper, etc.) 3 divs + 12 líneas de CSS keyframes. Cero JavaScript, cero dependencias.

6. Glosario

TérminoDefinición en el contexto del proyecto
SPASingle Page Application. Aplicación web de una sola página donde la navegación no recarga el documento.
ES6 ModulesSistema nativo de módulos de JavaScript con import/export, soportado desde 2017 en todos los navegadores modernos.
Event LoopMecanismo de JavaScript que procesa tareas de la cola de mensajes cuando el call stack está vacío, permitiendo comportamiento asíncrono.
